# AI cluster content generation, Auto Schema Generator, and Internal Link Health Score

Three features added to the ViraSEO WordPress plugin: AI-powered content generation for topical cluster pages, automatic JSON-LD structured data injection with an admin management UI, and a 5-factor internal link health scoring system with monthly trend tracking. The cluster content feature sends page context to an LLM and produces SEO-optimized Persian HTML with internal links, then allows the admin to review, edit, and apply it to posts. The schema generator detects page types (Article, Product, FAQ, HowTo, BreadcrumbList, Service, WebPage) and injects corresponding JSON-LD on the frontend, with collision detection for Yoast/RankMath/AIOSEO. The link health score computes a weighted composite from orphan ratio, average inlinks, link power distribution, broken links, and bidirectional coverage.

**Watch for:** AI-generated content is applied to live posts without rate limiting or undo UX beyond a single backup slot (confirmed). The Schema Generator's `other_schema_active()` check is shallow — it can false-negative on configured-but-inactive plugins (likely). The `loadLinkHealth()` call added to the scan callback will fire the health computation synchronously on every scan, which runs multiple full-table queries on the internal links table (confirmed).

## High-level view

The cluster content generation extends the existing InternalSilo class with three new AJAX endpoints: single-page generation, content application (with backup), and a batch eligibility check. The frontend drives a sequential generate-one-at-a-time loop rather than parallel requests, which avoids overwhelming the AI proxy but means generating content for 10 pages is a blocking UI wait with no cancellation mechanism.

The Schema Generator is a standalone class that hooks `wp_head` at priority 99 for frontend injection. Detection relies on keyword matching and HTML structure analysis of post content, with early-return logic that prevents combining schema types (e.g. a product with FAQ content only gets Product schema).

The link health score is a pure computation pass over the internal links table, producing a 0-100 score from five weighted factors. The computation is expensive (multiple aggregate queries) and runs 2-3 times per scan due to overlapping call paths.

1. **Single-slot content backup** — Applying AI content overwrites `_viraseo_content_backup` with no revision chain. A second apply loses the original. Store a revision counter or use WP's post revisions system.
2. **Schema collision false-negative** — `other_schema_active()` checks `WPSEO_VERSION` + `has_action('wpseo_json_ld')`, but Yoast may define the constant without registering that specific action. Switch to checking only the constant/class presence, not the action hook.
3. **Health computation on every scan** — `compute_link_health()` runs five aggregate SQL queries on potentially large tables. It's called synchronously in the scan response path and again via `record_link_health_snapshot()`. Cache the result in a transient or compute asynchronously.
4. **No AI cost cap** — Cluster content generation has no budget limit. Selecting 20 pages will fire 20 separate AI calls (4000 tokens each) without a confirmation threshold beyond the initial button click.
5. **FAQ extraction greedy match** — `build_faq()` grabs every h2/h3 + following paragraph pair as a Q&A, regardless of whether the heading is actually a question. This produces noisy FAQ schema for pages with regular subheadings.

## Cluster content generation lifecycle

The content-apply endpoint (`ajax_cluster_content_apply`) creates a backup in `_viraseo_content_backup` meta before overwriting. This is a single-slot backup — applying content a second time overwrites the backup with the already-AI-generated first version, losing the original human-written content (confirmed). WordPress native post revisions exist in the background (via `wp_update_post`), which partially mitigates this, but the backup meta gives users a false sense of a separate restore mechanism.

The meta description apply logic tries Yoast → RankMath → AIOSEO → own meta via an `elseif` chain, but it always also writes to `_viraseo_meta_desc` because the final `update_post_meta` call is outside the chain (confirmed). A Yoast site gets both `_yoast_wpseo_metadesc` and `_viraseo_meta_desc` set. Not blocking but could confuse debugging when meta descriptions appear to diverge.

The JavaScript generation loop (`generateNext`) is sequential-recursive with no cancellation. If the user navigates away or the tab sleeps, in-flight requests continue server-side while the UI state becomes stale.

## Schema detection and injection

Detection logic in `detect_schema_type()` returns early on first match: WooCommerce product → FAQ → HowTo → Service → Article/WebPage. A product page with FAQ content only gets Product+BreadcrumbList, never FAQPage. Google supports multiple schema types on a single page, so the early return loses potential rich results (likely, not blocking).

The `other_schema_active()` check for Yoast combines `defined('WPSEO_VERSION')` with `has_action('wpseo_json_ld')`. Yoast 20+ moved to `wpseo_schema_graph` filter instead, so the action check may fail. RankMath's `has_filter('rank_math/json_ld')` may similarly not be registered at `wp_head` priority 99 timing if RankMath uses a later priority. Either case results in duplicate schema injection on sites with active SEO plugins (likely).

`build_faq()` extracts every h2/h3 + immediately-following paragraph pair as a Question/Answer. On a typical blog post with 5 subheadings, this generates a FAQPage schema with 5 "questions" that aren't questions at all — just section headings. The detection gate (`has_faq_content`) only checks whether FAQ keywords exist or two headings end in `?`, but `build_faq` doesn't re-apply that question-mark filter when extracting pairs (confirmed).

## Link health score computation cost

`compute_link_health()` executes five queries including two `GROUP BY` aggregations on the internal links table. This method is called in `record_link_health_snapshot()` inside `scan()`, and the frontend also calls `loadLinkHealth()` immediately after scan completion (added in this diff to the scan callback). In practice the computation runs 2-3 times within seconds of each scan. For a site with thousands of internal links, caching the result in a transient for the duration of the request cycle would eliminate the redundancy.

## Link health history storage

The history stores up to 60 entries in a single `wp_options` row, each containing the full factors breakdown. At 60 entries this is ~15KB of autoloaded option data. On sites with `autoload='yes'` (the default for `update_option`), this adds to every page load's options query even though it's only consumed on the admin links page (possible performance concern on high-traffic sites).

## AI content prompt and URL trust

The page URLs passed to the AI prompt come from the frontend (`$_POST['page_url']`), which originates from cluster data computed via `get_permalink()` at an earlier point. The AI will embed these URLs as `<a href>` in the generated content. If a site changes its permalink structure between cluster computation and content generation, the resulting links will be stale. The content goes through `wp_kses_post` on apply, so the URLs won't be dangerous, but they may 404 (possible, low risk).
